# Project Maintenance Instructions

These instructions describe how to keep the project consistent. They are not measured results.

---

## Documentation Rules

- Do not invent accuracy, latency, memory, model-size, or dataset-size numbers.
- Use placeholders for runtime values until they are measured.
- Keep Android runtime documentation clear: the current app uses PyTorch Mobile/TorchScript assets.
- Keep ONNX/TFLite documentation clear: those artifacts are generated by the Python export pipeline and are not the current Android runtime path.
- When model assets change, update labels and metadata together.

Placeholder format:

```text
{{PLACEHOLDER_NAME}}
```

---

## Artifact Rules

Keep:

```text
SourceCode/models/best_model.pth
SourceCode/labels.txt
SourceCode/labels.json
SourceCode/metadata.json
agrilens/app/src/main/assets/plant_model.pt
agrilens/app/src/main/assets/labels.txt
```

Generated and removable when not needed:

```text
SourceCode/plant_model.onnx
SourceCode/plant_model.onnx.data
SourceCode/plant_model.pt
SourceCode/plant_model_tflite_float32/
SourceCode/plant_model_tflite_int8/
```

---

## Training Rules

- Run training from `SourceCode/`.
- Treat `configs/config.yaml` as the source of configuration truth.
- Preserve checkpoint class order.
- Preserve metadata and labels from the same run as the checkpoint.
- Record metrics only from the current run.

---

## Android Rules

- Keep `plant_model.pt` and `labels.txt` paired.
- Verify app build after UI, asset, or Gradle changes.
- Keep Vietnamese translations natural; keep technical English terms when translation would be unclear.
- Do not add treatment claims without a source or domain review.
